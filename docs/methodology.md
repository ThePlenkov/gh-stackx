# Stacked PR Methodology

This document explains the stacked-PR workflow that `gh-stackx` automates, and why it is useful even when GitHub's native Stacked PRs feature is not enabled.

## Why stack PRs?

A large change is hard to review all at once. Splitting it into a series of dependent pull requests keeps each review small and focused:

- Each PR is a single logical step.
- Reviewers can reason about one step before seeing the next.
- You can land the bottom PRs while the top ones are still being reviewed.

## Core concepts

- **Trunk**: the long-lived branch everything is based on, usually `main`.
- **Stack**: an ordered list of branches, where each branch is built on top of the previous one.
- **Bottom**: the branch closest to trunk.
- **Top**: the branch farthest from trunk.
- **Base**: the parent branch for a given layer. For the bottom layer the base is trunk; for every other layer it is the previous branch in the stack.

A typical stack looks like:

```
main
  └── feature/auth
        └── feature/api
              └── feature/ui
```

In this example:

- `feature/auth` is based on `main`.
- `feature/api` is based on `feature/auth`.
- `feature/ui` is based on `feature/api`.

## Workflow overview

1. **Create the stack locally**  
   Use `gh stackx init` and `gh stackx add` to build the stack.

2. **Work on each layer**  
   Navigate with `gh stackx up` / `gh stackx down`. Commit changes on the current branch.

3. **Push and open PRs**  
   Run `gh stackx submit`. It pushes the branches and creates or updates PRs with the correct `--base` and `--head` values. PRs are created as drafts by default; pass `--open` to mark them ready.

4. **Sync after upstream changes**  
   When `main` or a parent PR moves, run `gh stackx sync` to rebase the stack and update each PR's base branch.

5. **Merge from the top down**  
   Run `gh stackx merge`. The top PR is merged first, then the next one down, and so on. Each merge updates the base of the next PR automatically as the branches are merged into the trunk.

## Why `gh-stackx` instead of `gh stack submit`?

GitHub's Stacked PRs API is in private preview. On most repositories `gh stack submit` fails with:

```
⚠ Stacked PRs are not enabled for this repository
```

`gh-stackx` solves this by using standard `gh pr` commands. It still uses `github/gh-stack` for local operations, so the local stack metadata (`.git/gh-stack`) is identical.

## Important rules

- **Keep the working tree clean before `sync` or `merge`.** Rebase operations will fail if there are uncommitted changes.
- **Create PRs bottom-up.** Each PR must point to the correct base, and the base must already exist on the remote.
- **Merge top-down.** This ensures the dependent PRs are merged before their parents change.
- **Do not commit `.git/gh-stack`.** That file is local metadata and will be regenerated by `gh stack view`.

## When the native feature is enabled

If your repository later gains the GitHub Stacked PRs feature, you can switch to `gh stack submit` and `gh stack merge`. Until then, `gh-stackx` provides the same end-to-end workflow.
